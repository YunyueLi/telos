# Strategy

Telos should be useful as a product and reusable as an engine.

## Product Principle

The public repository is the canonical project entry point. It should show the actual product, run locally, and make the core workflow understandable without requiring access to hosted infrastructure.

## Learning Model

Telos classifies learning work by mechanism rather than school subject:

| Class | Examples | Main mechanism |
| --- | --- | --- |
| A | vocabulary, terminology, law, anatomy | declarative memory and retrieval |
| B | math, programming, algorithms | well-structured procedural learning |
| C | writing, design, research | ill-structured creative work |
| D | music, sports technique, craft | closed-loop physical skill |
| E | esports, debate, competitive domains | open and adversarial performance |
| F | habits, leadership, emotional regulation | affective and behavioral change |

The engine is strongest today for A/B-style graphs and already has room in the data model for broader domains.

## Interface Direction

Learning maps are not decorative. They are the main interface for:

- showing why the next node is recommended
- separating mastered, learning, frontier, and locked nodes
- keeping diagnosis and review connected to the graph
- making the learner's progress inspectable

The Community Edition should keep the app polished and runnable while leaving managed hosted operations outside the public repository.

## Community Direction

Good public contributions make Telos easier to learn from, run, and extend:

- better examples
- clearer docs
- stronger engine tests
- higher-quality graph normalization
- accessible, responsive product states
- interoperable schema and export formats

The project should grow recognition through a strong canonical public repo, not through a split core-only shell.
